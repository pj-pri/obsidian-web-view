# Security / Auth Follow-ups

세션 로그인 구조는 구현되었지만, 아래 항목들은 **운영 전 적용 또는 검증이 필요한 후속 작업**입니다.

이 문서는 “지금 당장 꼭 구현해야 하는 것”보다, **나중에 잊기 쉬운 운영/보안 체크리스트**를 남기는 용도입니다.

---

## 1. 운영 전 반드시 적용

### 1.1 `SESSION_SECRET` 실제 랜덤 값으로 교체

현재 로컬 `.env`에는 예시 문자열이 들어갈 수 있습니다.

- 해야 할 일:
  - `SESSION_SECRET=replace-with-a-long-random-secret`
  - 같은 예시값을 **실제 랜덤 문자열**로 교체

- 권장:
  - 길고 예측 불가능한 문자열 사용
  - 예: 32바이트 이상 랜덤 값

### 1.2 기본 계정값 강하게 변경

- `APP_LOGIN_USER`
- `APP_LOGIN_PASSWORD`

예시/약한 값을 그대로 두지 말고 실제 운영용 값으로 교체해야 합니다.

### 1.3 외부 공개 시 HTTPS 뒤에서 운영

현재는 HTTP에서도 동작하지만, 외부 공개 시에는 반드시 reverse proxy + HTTPS 뒤에서 운영해야 합니다.

이유:
- 로그인 정보 보호
- 세션 쿠키 보호
- 추후 `Secure` 쿠키 적용 기반 확보

---

## 2. HTTPS 적용 후 바로 같이 할 것

### 2.1 세션 쿠키 `Secure` 적용

세션 로그인은 현재 `HttpOnly`, `SameSite=Lax`를 사용합니다.

HTTPS가 준비되면 다음을 검토/적용:
- 쿠키에 `Secure` 추가

### 2.2 reverse proxy 가이드 문서화

아직 남아 있는 문서 작업:
- nginx 예시
- caddy 예시
- `X-Forwarded-For` 전달 구성 점검

---

## 3. 운영 전 수동 검증 권장

### 3.1 브라우저 UX 수동 확인

아직 자동 검증만 했고, 브라우저에서 아래 항목을 직접 확인하는 것이 좋습니다.

- 로그인 화면 표시
- 로그인 성공 후 vault 진입
- 로그아웃 버튼 동작
- 로그인 실패 메시지 표시
- 세션 만료/401 발생 시 로그인 화면 복귀

### 3.2 세션 유지 시간 확인

`SESSION_TTL_MS` 동작을 실제로 확인하는 테스트는 아직 충분히 하지 않았습니다.

확인할 것:
- 로그인 직후 정상 접근
- TTL 경과 후 재로그인 요구 여부
- 요청 중 활동 시 세션 연장 체감 여부

---

## 4. 현재 구조의 한계

### 4.1 메모리 기반 세션 저장소

현재 세션은 서버 메모리에 저장됩니다.

영향:
- 서버 재시작 시 전체 로그아웃
- 여러 인스턴스 운영에 부적합

지금 단계에서는 허용 가능하지만, 나중에 확장 시 고려:
- 공유 저장소 기반 세션
- 토큰 기반 인증

### 4.2 메모리 기반 rate limiting

현재 request/auth limiter도 단일 프로세스 메모리 기반입니다.

영향:
- 단일 인스턴스 self-hosted에는 충분
- 다중 인스턴스/분산 환경에서는 불충분

---

## 5. 향후 인증 고도화 후보

아직 미구현:

- 토큰 기반 인증
- 다중 사용자 구조
- 관리자/읽기전용 등 권한 분리
- 감사 로그 고도화

현재 프로젝트 성격상 우선순위는 낮지만, 아래 상황이면 검토 가치가 커집니다.

- 여러 사용자가 접근해야 함
- 앱 내부 권한 분리가 필요함
- 외부 인터넷 공개 운영을 본격화함
